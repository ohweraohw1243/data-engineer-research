# Внешний источник задач: подробный план и большой промт

## 1) Что нужно для внешнего источника задач

1. Хранилище контента.
- Вариант A: GitHub (репозиторий с JSON-файлами) + CDN ссылка.
- Вариант B: Supabase Storage.
- Вариант C: Cloudflare R2.
- Для старта самый простой путь: GitHub + CDN.

2. Формат данных.
- Один манифест с версией и списком секций.
- Отдельный JSON на каждую секцию: coding, interview, math, questions.
- У каждой задачи обязательные поля: id, stage, type, prompt, answer.
- Дополнительные поля: hints, tags, difficulty, source, updatedAt.

3. Логика загрузки.
- Открывается раздел.
- Приложение сначала проверяет локальный кэш.
- Если кэш свежий, показывает его сразу.
- Параллельно тихо проверяет обновления на удаленном источнике.
- Если удаленный источник недоступен, продолжает работать на кэше или встроенном локальном наборе.

4. Кэш и отказоустойчивость.
- TTL кэша, например 6-24 часа.
- Версия схемы данных (schemaVersion), чтобы мигрировать старые данные.
- Fallback на локальные данные, чтобы сайт никогда не оставался пустым.

5. Безопасность и качество.
- Все внешние тексты считать недоверенными, не вставлять как HTML.
- Валидация JSON перед использованием.
- Логи ошибок загрузки.
- Ограничение времени запроса (timeout), например 3-5 секунд.

6. Что может усложнить внедрение.
- CORS на стороне источника.
- Лимиты запросов.
- Нестабильные ссылки без стабильного CDN.
- Неединообразный формат задач при ручном редактировании.

## 2) Оценка трудозатрат

1. V1 (минимально рабочий, надежный): 3-6 часов.
2. V2 (фильтры, теги, приоритеты, A/B банка): 1-2 дня.
3. V3 (редактор задач, модерация, аналитика): 3-7 дней.

## 3) Рекомендуемый путь

1. Сделать V1 через удаленный JSON-манифест + секционные JSON.
2. Оставить текущие локальные данные как аварийный fallback.
3. Добавить простую админ-процедуру: изменил JSON в репозитории, сайт подтянул обновления автоматически.

## 4) Большой промт для отдельного обсуждения с другой нейросетью

Ты senior solution architect и frontend engineer. Нужна детальная архитектура для статического учебного сайта по Data Engineering, где задачи сейчас локальные, а мы хотим грузить их по ссылке из внешнего источника (без тяжелого backend на старте).

Контекст проекта:
- Это статический сайт (HTML/CSS/JS).
- Есть несколько разделов с задачами: coding, interview, math, questions.
- Есть прогресс пользователя и логика проверки ответов.
- Нужна высокая надежность: если сеть недоступна, сайт не должен ломаться.
- Язык интерфейса: русский.
- Хотим масштабировать банк задач до большого объема.

Что нужно от тебя:
1. Предложи 2-3 архитектурных варианта:
- Удаленный JSON через CDN.

- Полноценный API.
Для каждого дай плюсы, минусы, риски, стоимость поддержки, сложность внедрения.

2. Дай рекомендуемый вариант для старта и обоснуй, почему именно он.

3. Спроектируй структуру данных:
- Манифест банка задач.
- Формат секционных файлов.
- Обязательные и опциональные поля задач.
- Версионирование schemaVersion.
- Миграция старых данных.

4. Опиши клиентский алгоритм загрузки:
- Cache-first + stale-while-revalidate.
- Таймауты.
- Fallback-стратегия при падении источника.
- Поведение при битом JSON.

5. Дай конкретные рекомендации по безопасности:
- Санитизация контента.
- Ограничения на рендер.
- Защита от XSS через контент задач.
- Валидация схемы данных на клиенте.

6. Опиши требования к инфраструктуре:

- Как решить CORS.

7. Дай пошаговый план внедрения:

8. Дай тест-план:

10. В конце выдай:
