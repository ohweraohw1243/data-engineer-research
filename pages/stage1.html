Этап 1: Теория СУБД, Архитектура и SQL

Исчерпывающий теоретический фундамент. В этом разделе собраны подробные архитектурные концепты, знания о внутренних механизмах PostgreSQL и ClickHouse, которые необходимы для успешного прохождения секции Data Architecture и SQL в крупных компаниях.

Теоретические модули

📚 Модуль 1: Эволюция аналитических хранилищ

Чтобы успешно мигрировать пайплайны и строить платформу, важно понимать, почему архитектуры исторически развивались от классических баз до Data Lakehouse.

1. DWH vs Data Lake vs Data Lakehouse

DWH (Data Warehouse): Традиционное хранилище структурированных данных (например, Teradata, Oracle, Greenplum). Базируется на парадигме Schema-on-Write: схема данных (таблицы, типы столбцов) жестко определяется до загрузки. Это гарантирует высокое качество данных и скорость SQL-запросов, но делает систему дорогостоящей в масштабировании и негибкой для неструктурированных данных (ML, логи, JSON). DWH часто использует ACID-транзакции и глубоко завязан на вычислительные мощности (состояние compute and storage tightly coupled).

Data Lake (Озеро данных): Ответом на дороговизну DWH стал Hadoop (HDFS). Парадигма Schema-on-Read: данные (логи, картинки, CSV, JSON) "сваливаются" в дешевое распределенное хранилище как есть. Схема накладывается только в момент чтения (например, через Spark или Hive). Плюсы: бесконечное и дешевое горизонтальное масштабирование (Commodity Hardware). Минусы: отсутствие транзакционности, невозможность нормального UPDATE/DELETE, риск превращения в "Data Swamp" (болото данных) из-за потери контроля над качеством и метаданными.

Data Lakehouse: Современная архитектура, объединяющая гибкость и дешевизну Data Lake с транзакционностью и структурой DWH. Хранение остается дешевым (объектные хранилища S3), но поверх него накладывается ACID-слой транзакций и метаданных (форматы таблиц: Apache Iceberg, Delta Lake, Apache Hudi). Это позволяет делать SQL UPDATE/DELETE, путешествовать во времени (Time Travel) и обеспечивать строгую консистентность прямо поверх "сырых" Parquet-файлов в S3, разделяя хранение (Storage) и вычисления (Compute — Spark, Trino, ClickHouse).

2. Основные парадигмы: Schema-on-Write vs Schema-on-Read

Schema-on-Write (Схема при записи): Перед вставкой данных реляционная БД или DWH проверяет их на соответствие жесткой схеме (типы данных, ограничения, внешние ключи). Если в JSON-источнике появилось новое поле или изменился тип (например, int стал string) — ETL-пайплайн падает, защищая DWH от мусора. Это дает 100% гарантию структуры для аналитиков, но требует постоянной поддержки (DDL-миграций) со стороны инженеров.

Schema-on-Read (Схема при чтении): Характерна для Data Lake. Мы сохраняем сырые файлы (например, JSON-события) в S3 без валидации типов на входе ("закидываем как есть"). Когда аналитик пишет SELECT * FROM s3_table, движок (например, AWS Athena или Spark) применяет схему "на лету", пытаясь распарсить файл согласно заданным в метасторе (Hive Metastore) правилам. Если структура файла не совпадает со схемой — мы получаем NULL в столбцах или ошибку чтения, но сам процесс сохранения сырых данных никогда не блокируется.

3. HDFS vs S3 (и объектные хранилища)

HDFS (Hadoop Distributed File System): Классическая распределенная файловая система. Имеет строгую иерархию папок, поддерживает Append (дозапись в файл). Вычисления в HDFS (MapReduce) работают по принципу Data Locality: вычисления отправляются на те же физические узлы (Data Nodes), где лежат блоки данных. HDFS отлично работал в on-premise кластерах, но имеет предел масштабирования из-за NameNode (который хранит все метаданные в RAM) и жесткой связки дисков и процессоров.

S3 (Simple Storage Service): Объектное хранилище (Object Store). В нем нет реальных папок и файлов, есть только "Бакеты" (Buckets) и плоский список "Объектов" с ключами (например, bucket-name/folder/data.parquet — это просто уникальная строка-ключ). В S3 нет Data Locality (вычисления и хранение живут в разных сетях), файлы иммутабельны (нельзя дописать в середину объекта, только перезаписать целиком). Зато S3 предлагает бесконечное, феноменально дешевое масштабирование без узкого горлышка NameNode, что сделало его индустриальным стандартом для Data Lakehouse (как в облаке AWS, так и on-premise через MinIO или Ceph RADOS Gateway).

4. Parquet: Внутреннее устройство и Колоночное хранение

В строковых форматах (CSV, JSON, классический PostgreSQL) данные физически лежат на диске строка за строкой. Если в таблице 100 колонок, а запрос SELECT user_id, amount обращается к двум, диск все равно прочитает все 100 столбцов (Full Scan). Apache Parquet — это бинарный колоночный формат. Значения одного столбца хранятся на диске рядом друг с другом.

Внутренняя иерархия Parquet:

  • Файл (File): Состоит из магии (Magic Bytes: 'PAR1'), одной или нескольких Row Groups и Footer-а. Важно: все метаданные лежат в Footer (в конце файла). Читатель (Spark) сначала прыгает в конец файла, читает Footer и понимает, где лежат нужные колонки.
  • Row Group (Группа строк): Горизонтальный срез данных (например, 1 миллион строк или 128 МБ). Каждая Row Group содержит все колонки для этих строк.
  • Column Chunk (Чанк колонки): Кусок конкретной колонки внутри одной Row Group. Здесь лежат сами данные.
  • Page (Страница): Наименьшая единица чтения (обычно 1-8 МБ). Страницы сжимаются и кодируются.

Ключевые механизмы ускорения в Parquet:

  • Кодирование (Encoding): Так как в колонке лежат однотипные данные (например, миллион нулей и единиц), их легко сжать. Применяются: Dictionary Encoding (замена длинных повторяющихся строк на короткие ID) и RLE (Run-Length Encoding) (замена "0,0,0,0,0" на "5 x 0").
  • Predicate Pushdown (Фильтрация на уровне хранилища): В Footer файла и в метаданных каждой Row Group/Page хранятся статистики: минимум (MIN) и максимум (MAX) для каждой колонки. Если мы делаем SELECT * WHERE age > 50, механизм проверяет Footer: если max_age в этой Row Group равно 40, весь многомегабайтный блок даже не читается с диска. Это радикально ускоряет I/O.
  • Projection Pushdown: Spark извлекает из Parquet только те Column Chunks, которые явно указаны в SELECT, полностью игнорируя остальные столбцы на диске.

🐘 Модуль 2: Внутренности PostgreSQL и Оптимизация PostgreSQL

Даже если основной стек — ClickHouse и Spark, PostgreSQL остается мета-базой и сервисной БД для тысяч внутренних сервисов.

1. MVCC (Multiversion Concurrency Control)

PostgreSQL не перезаписывает строки физически при UPDATE. Он помечает старую версию строки как "мертвую" (записывая ее xmax), и создает новую версию (с новым xmin). Это позволяет пишущим транзакциям не блокировать читающие. Но это порождает проблему: мертвые строки копятся (bloat). Для их очистки работает фоновый процесс VACUUM. Если он настроен плохо, наступает коллапс диска.

2. Изоляция транзакций (ACID)

Уровни изоляции: Read Uncommitted (в PG работает как Read Committed), Read Committed (по умолчанию, видит только закоммиченные данные до начала стейтмента - возможно несовпадение данных при повторном чтении), Repeatable Read (сохраняет снимок БД на начало транзакции) и Serializable (жесткие блокировки, имитация последовательного выполнения). Нужно понимать разницу между аномалиями: Dirty Read, Non-repeatable read и Phantom Read.

3. Индексы: Под капотом B-Tree

Физически это сильно-ветвистое дерево, где все данные лежат в листьях на одном уровне глубины. При запросе с EXPLAIN важно отличать Index Scan (Сначала БД нашла строку в индексе, потом сходила на диск в "кучу" Heap, чтобы достать те колонки, которых в индексе нет) от Index-Only Scan (Запрос попросил только те колонки, которые уже включены во вторичный индекс на этапе создания ключа - это в разы быстрее, так как не надо лезть в "кучу").

4. Оконные функции (SQL)

В отличие от GROUP BY, окна не схлопывают строки. Понимание фреймов (Framing): ROWS задает физические границы (2 строки назад), RANGE — логические (по стоимости сортировки). Дефолтный фрейм при ORDER BY — это RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW, что при наличии дублей может "повесить" запрос (O(n²)). Разница: ROW_NUMBER() просто инкрементирует счетчик, RANK() пропускает места при совпадении (1, 2, 2, 4), а DENSE_RANK() оставляет места подряд (1, 2, 2, 3).

🚀 Модуль 3: ClickHouse: выжимаем OLAP-максимум

ClickHouse будет вашим основным OLAP движком в новой дата-платформе. Это колоночная СУБД с принципиально иными механизмами работы в сравнении с PostgreSQL.

1. Движки семейства MergeTree

Данные пишутся пачками (parts), которые сортируются в момент попадания на диск, а затем в фоне процессы ClickHouse сливают (merge) эти пачки вместе. Настоящих операций UPDATE и DELETE здесь нет (они эмулируются тяжелым мутационным запросом ALTER). Для реальной работы с изменяемыми данными созданы CollapsingMergeTree (со столбцом Знака +1/-1) и ReplacingMergeTree (оставляет самую свежую версию по версии времени).

2. Primary Key vs Order By и Разреженный индекс

В отличие от PostgreSQL, PRIMARY KEY в ClickHouse не гарантирует уникальность. Он выполняет роль разреженного индекса (Sparse Index). Данные на диске нарезаны на гранулы (по умолчанию 8192 строки). Первая строка каждой гранулы попадает в индекс. Когда вы делаете фильтрацию по первичному ключу, база быстро понимает, в каких гранулах поискать данные, а какие пропустить целиком (skip granules).

3. Партиционирование и Шардирование

  • Партиционирование (PARTITION BY): Обычно по месяцам или дням. Разделяет данные на независимые куски на одном сервере. Отлично подходит для быстрого удаления старых данных (ALTER TABLE ... DROP PARTITION).
  • Шардирование (Distributed таблица): Режет данные по хэш-функции от какого-то ключа (например user_id) на разные сервера кластера, балансируя нагрузку и объем дисков.

Проверка усвоения теории (Quiz)

Ниже представлены 5 теоретических вопросов, которые часто задают мидлам на секциях Data Architecture / DB Internals. Проверьте себя и нажмите "Показать ответ", чтобы свериться с правильным подходом.

Вопрос 1: Парадигмы хранилищ

В чем техническое ключевое различие парадигм "Schema-on-Write" и "Schema-on-Read", и к каким архитектурам (DWH, Data Lake) они относятся? В чем слабое место Озера Данных?

Schema-on-Write (DWH): Схема данных (типы колонок, ключи) жестко задается до записи (например, в PostgreSQL/Greenplum). Если данные не подходят под схему, запись падает с ошибкой. Плюс: строгая типизация и быстрое чтение.

Schema-on-Read (Data Lake): Данные сохраняются в сыром виде в распределенном хранилище (JSON на S3/HDFS). Схема накладывается только в момент чтения (через Spark/Presto). Из-за этого данные могут быть мусорными.

Слабое место Data Lake — без жесткого управления (Data Governance/Catalog) озеро быстро превращается в "болото данных" (Data Swamp).

Вопрос 2: Внутренности PostgreSQL (MVCC)

Объясните, почему в PostgreSQL таблица может раздуваться в размерах даже если вы не вставляете новые данные, а только обновляете существующие (делать частый UPDATE). Как работает процесс VACUUM?

Из-за механизма MVCC (Multiversion Concurrency Control) операция UPDATE в PostgreSQL физически работает как DELETE старой версии строки (тупла) и INSERT новой версии. Старая строка помечается как "мертвая" (через заголовок xmax), но остается на диске, так как ее могут пока еще читать конкурентные паралелльные транзакции.

Фоновый сборщик мусора (VACUUM) проходит по страницам диска и ищет такие "мертвые" туплы, помечая занимаемое ими пространство как свободное (Free Space Map) для последующих вставок. Без работы VACUUM файл таблицы будет бесконечно пухнуть.

Вопрос 3: Оптимизация СУБД (Scan Types)

Вы анализируете план запроса (EXPLAIN) и видите Index Scan, а затем переписываете запрос и видите Index-Only Scan. В чем заключается их архитектурная разница, и почему второй радикально быстрее?

Index Scan: БД находит нужный указатель(типа RowID) в индексе (B-Tree), но требуемых колонок для вывода (SELECT cols) в индексе нет. Поэтому БД делает дополнительное дисковое чтение страницы из табличной "кучи" (Heap), чтобы достать недостающие колонки.

Index-Only Scan: Запрос составлен так, что все запрашиваемые колонки уже присутствуют прямо в самом индексе (как ключ или через INCLUDE). БД возвращает результат из кэша дерева индекса, вообще не обращаясь к куче (при условии что Visibility Map позволяет), что в разы снижает медленный дисковый I/O.

Вопрос 4: Под капотом ClickHouse

В ClickHouse (MergeTree) Primary Key не контролирует уникальность строк (как это принято в классических БД). Для чего тогда он нужен, что такое "гранулы" и как он помогает ускорить SELECT-запросы?

В движке MergeTree данные лежат физически отсортированно (по столбцам из ORDER BY) и нарезаны на блоки, называемые гранулами (по умолчанию 8192 строки).

Primary Key формирует "разреженный индекс" (sparse index) — он хранит значение ключа не для физически каждой строки (как B-tree), а только для первой(стартовой) строки каждой гранулы. Когда приходит SELECT с WHERE-фильтром, ClickHouse загружает в оперативку первичный ключ с диска, применяет бинарный поиск и за доли миллисекунд выбрасывает гранулы вне диапазона. Грузятся только нужные куски, что колоссально экономит время.

Вопрос 5: SQL Оконные функции

Вы пишете запрос SUM(amount) OVER (PARTITION BY user_id ORDER BY purchase_date). Каким является фрейм (окно строк) по умолчанию, и почему наличие полных дублей purchase_date может привести к неожиданному результату или сильному повешению памяти (O(n²))?

По умолчанию при указании ORDER BY в PostgreSQL граница фрейма выставляется неявно в: RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW.

RANGE обрабатывает не физические скачки, а логические значения ключа сортировки. Это значит что он стягивает все строки с одинаковым значением сортировки (дубли дат) в один блок. Если у одного юзера 10 000 покупок в одну и ту же дату без времени (микросекунд), база будет вынуждена держать их все в памяти для пересчета агрегата, что ведет к деградации до O(n²).

Чтобы считать агрегат строго строка за строкой за O(N), нужно явно сказать БД считать по физическим строкам: ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW.
