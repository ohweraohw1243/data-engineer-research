🐘 Тема 1: Архитектура DWH, Lake и OLAP

Суть под капотом

DWH vs Data Lake: DWH (Data Warehouse) хранит структурированные данные (Схема при записи - Schema on Write). Data Lake (S3, HDFS) хранит сырые данные (Схема при чтении - Schema on Read). В SMT мы мигрируем пайплайны с HDFS на S3 для создания Data Lake.

Моделирование данных: Звезда (Star Schema) и Снежинка. Разница между подходами Ральфа Кимбалла (шины данных и витрины) и Инмона (единое нормализованное хранилище). Data Vault — гибрид для аудита и историчности с разделением на Hub, Link, Satellite.

Хранение в S3: Использование партиционирования путей (например, s3://bucket/topic/year=2026/month=04/) и форматов файлов как Parquet/ORC для колоночного чтения.

🚀 Тема 2: ClickHouse (Семейство MergeTree)

Суть под капотом

MergeTree: Это не просто таблица, а целое семейство движков. Данные записываются кусками (parts), а в фоновом режиме сливаются (merge). Нет Update/Delete (используем CollapsingMergeTree или ReplacingMergeTree).

Primary Key vs Order By: В ClickHouse Primary Key не гарантирует уникальность. Он создает sparse-индекс (разреженный), который позволяет быстро сужать поиск гранул данных (granules) при фильтрации.

Шардирование и партиционирование: Партиционирование (обычно по месяцам) для ротации и DROP старых данных. Шардирование для распределения нагрузки по кластеру (Distributed таблицы).

🔍 Тема 3: SQL На максималках

Суть под капотом

Оконные функции и CTE: Окна избавляют от self-join. Понимание разницы между ROWS и RANGE предел (frame semantics). Рекурсивные CTE (WITH RECURSIVE) для работы с деревьями и иерархиями.

Под капотом PostgreSQL: Поиск узких мест с EXPLAIN ANALYZE. Какие бывают Join: Nested Loop, Hash Join, Merge Join. Влияние буферов (work_mem) на производительность сортировок.

Индексы: B-tree, Hash, GIN. Разница между Index Scan (чтение индекса + поход в кучу Heap) и Index-Only Scan (данные читаются прямо из листьев индекса, если таблица VACUUM-обработана).

Интерактивная практика (Оконные)

Задача: Напишите SQL, который для каждого user_id из таблицы purchases (user_id, purchase_date, amount) находит день его первой покупки и сумму, потраченную в этот день (через оконную функцию).
Используйте ROW_NUMBER() OVER(PARTITION BY user_id ORDER BY purchase_date ASC). Затем отфильтруйте внешним запросом rn = 1.
WITH RankedPurchases AS ( SELECT user_id, purchase_date, amount, ROW_NUMBER() OVER(PARTITION BY user_id ORDER BY purchase_date ASC) as rn FROM purchases ) SELECT user_id, purchase_date, amount FROM RankedPurchases WHERE rn = 1;
