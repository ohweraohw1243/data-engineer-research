📅 День 1: Генераторы и управление памятью

Доступное время

5 часов

Теория "под капотом"

Lazy evaluation: yield приостанавливает выполнение функции, возвращая значение вызывающей стороне, и оставляет локальные переменные в памяти виртуальной машины Python. При вызове next() выполнение возобновляется строго с места прошлого yield.

Архитектура памяти CPython: В стандартном интерпретаторе Python генераторы задействуют инструкции байткода YIELD_VALUE и обертку объекта Generator Object. Поскольку в памяти держится только состояние функции, а не результирующая коллекция (что делает List Comprehensions), сложность по памяти падает с O(n) до строго O(1).

Лайв-кодинг: Интерактивная практика

Задача 1: Чтение гигантского файла
Дан файл-лог размером 10GB. Напишите генератор read_large_file(filepath), который открывает файл и возвращает строки по одной, чтобы потребление RAM оставалось константным.
Подсказка: Файловые объекты в Python сами по себе являются итераторами. Используйте обычный цикл for line in f: и yield line внутри.
with open(filepath, 'r', encoding='utf-8') as f: for line in f: yield line.strip()

📅 День 2: Chunking (Пакетная обработка)

Доступное время

3 часа
13:00 - 16:00 (Среда, четная неделя)

Теория "под капотом"

Оптимизация работы с БД и I/O: Отправлять в PostgreSQL строки по одной (one-by-one insert) — критически медленно из-за накладных расходов на сетевые RTT (Round Trip Time) пакеты протокола. Использование "chunking" (формирование батчей по 1000–5000 штук) является стандартом. Библиотека psycopg2 предоставляет метод execute_values(), который трансформирует массив кортежей в единый многострочный INSERT ... VALUES (),().

Лайв-кодинг: Интерактивная практика

Задача 2: Генератор батчей
Напишите функцию chunker(iterable, size) на Python. Она должна принимать любой итератор и размер батча (size), и возвращать список элементов длиной size на каждой итерации. (Можно использовать itertools.islice).
from itertools import islice def chunker(iterator, size): iterator = iter(iterator) while True: chunk = list(islice(iterator, size)) if not chunk: break yield chunk

📅 День 3: Multiprocessing

Доступное время

2 часа
09:00 - 11:00 (Четверг, четная неделя)

Теория "под капотом"

GIL (Global Interpreter Lock): Исторический архитектурный мьютекс в CPython. Он запрещает параллельное исполнение байткода Python несколькими нативными потоками OS внутри одного процесса. Из-за этого модули 'threading' в Python помогают только при I/O задачах (где поток засыпает в ожидании сети или диска, освобождая GIL).

Модуль multiprocessing: Создает честные отдельные процессы ОС со своим выделенным адресным пространством и своим интерпретатором. Это позволяет утилизировать все ядра CPU для тяжелых вычислений (CPU-bound). Минус — дорогое создание и сериализация (через pickle) при передаче данных между процессами.

📅 День 4-5: asyncio & asyncpg

Доступное время

15 часов
Пт: 15:00-19:00 | Сб: 9-11, 19-22 | Вс: 10-16

Теория "под капотом"

Event loop (Цикл событий): В основе asyncio лежит невытесняющая кооперативная многозадачность. Одна нить выполнения крутит цикл (event loop), который опрашивает открытые сокеты ОС (используя epoll в Linux или kqueue). Когда вы пишете await db.fetch(), ваша сопрограмма (coroutine) говорит циклу: "Отдай управление другим задачам, пока сокет базы данных мне не ответит".

asyncpg: Это самый быстрый драйвер PostgreSQL для Python, написанный на Cython. Он не использует стандартную libpq. Он упаковывает данные напрямую в бинарный протокол Postgres 3.0 без лишних аллокаций. Результат — скорость выгрузки/записи выше стандартного psycopg2 в 2–3 раза для I/O heavy задач.
