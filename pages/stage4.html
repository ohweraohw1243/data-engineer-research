Теоретические модули

🌊 Модуль 1: Архитектура Apache Kafka

Apache Kafka — это распределенный брокер сообщений. Он используется для потоковой (Streaming) передачи данных в реальном времени.

  • Топики и Партиции: Данные пишутся в Topics (как таблицы), которые делятся на Partitions для параллельной записи и чтения.
  • Producer & Consumer: Продюсеры отправляют данные, а Консюмеры (объединенные в Consumer Groups) читают их. Kafka просто хранит лог событий (Append-only log).
  • Гарантии доставки: At-most-once (можем потерять), At-least-once (базовый стандарт, могут быть дубли), Exactly-once (идеально, но сложно в настройке).

🧬 Модуль 2: Data Governance и Data Lineage

Когда хранилище разрастается до сотен таблиц, нужно понимать, откуда берутся данные и кто ими владеет.

  • Data Catalog: Единый реестр (словарь) всех таблиц, метрик и источников в компании (например, DataHub, Amundsen, OpenMetadata).
  • Data Lineage: Визуальный граф "происхождения" данных. Показывает, из какой сырой таблицы (STG) собралась витрина (DDS) и в каком BI-дашборде она используется. Помогает при расследовании инцидентов.

Что посмотреть/почитать:

  • ▶️ YouTube: Что такое Data Governance простыми словами.
  • 📝 Статья: Внедрение DataHub или OpenMetadata (как работает Lineage).

🛡️ Модуль 3: Data Quality (Качество данных)

Хороший пайплайн не просто перекладывает данные, но и проверяет их на аномалии.

  • Data Contracts: Соглашение (контракт) между поставщиком данных (бэкенд) и потребителем (DWH). Если бэкенд меняет тип колонки — пайплайн падает еще на этапе CI/CD.
  • Проверки (DQ Checks): Проверка на NULL, уникальность (Primary Key), диапазоны значений (например, возраст > 0). Инструменты: Great Expectations, Soda.
  • Мониторинг/Алертинг: Если в таблице стало на 50% меньше строк, чем вчера — отправляем алерт в Slack/Telegram.

  • 💻 Инструментарий: Документация Great Expectations (как писать базовые тесты для данных).
