Этап 5: Системный дизайн (System Design)

Проектирование и архитектура платформ данных. Как выбрать между Batch и Streaming, проектирование DWH и Data Lake. Все практические задания перенесены в раздел "Лайв-кодинг".

Теоретические модули

🧩 Модуль 1: Архитектура DWH, Data Lake и Lakehouse

Эволюция систем хранения аналитических данных:

  • Data Warehouse (DWH): Структурированные данные (SQL), жесткая схема (Schema-on-Write). Примеры: ClickHouse, Greenplum, Snowflake. Отлично для готовой отчетности.
  • Data Lake: Хранение сырых данных любых форматов (JSON, Parquet, картинки) в дешевом облаке (S3/HDFS). Гибкость на старте, но есть риск превратить хранилище в "болото данных" (Data Swamp).
  • Data Lakehouse: Гибрид. Данные лежат дешево в S3, но поверх них используются ACID-транзакции и SQL-движки (как в DWH). Популярные форматы: Apache Iceberg, Delta Lake, Apache Hudi.

  • 📖 Книга: "Designing Data-Intensive Applications" (Martin Kleppmann) — Библия архитектуры (Глава 3 про Storage Engine).
  • ▶️ YouTube: Data Lake vs Data Warehouse vs Data Lakehouse (концептуальные основы).
  • 📝 Статья: Введение в Apache Iceberg для Data Engineer (как устроено версионирование).

⚖️ Модуль 2: Lambda и Kappa Архитектуры

Две классические парадигмы проектирования пайплайнов для обработки потоковых и исторических данных:

  • Lambda Architecture: Данные делятся на 2 пути: Batch Layer (считает все точно по большим массивам, но раз в сутки) и Speed Layer (считает быстро в рантайме онлайн). Serving Layer объединяет результаты для пользователя (точность + скорость).
  • Kappa Architecture: Упрощение лямбды. Разделения на Batch нет! Все данные обрабатываются только как потоковые (Speed Layer), даже исторические данные, если нужно пересчитать (например Kafka + Flink).

Что посмотреть/почитать:

  • ▶️ YouTube: Паттерны проектирования потоковой обработки данных на масштабах.

🗂️ Модуль 3: Моделирование данных (Data Modeling)

Как физически и логически располагать таблицы в DWH, чтобы BI-дашборды крутились быстро, а аналитикам было удобно:

  • Нормализация (3NF): Минимизация дублирования данных (Inmon). Идеально для транзакционных баз (OLTP), но тяжело для аналитики (много JOIN).
  • Star Schema (Звезда) / Snowflake: Подход Kimball (Dimensional Modeling). Одна центральная таблица фактов, окруженная маленькими таблицами измерений (справочниками). Оптимально для аналитики (OLAP).
  • Data Vault: Архитектура под agile-хранилища (Хабы, Линки, Сателлиты). Позволяет легко расширять исторические модели баз.

Что посмотреть/почитать:

  • 📖 Книга: "The Data Warehouse Toolkit" (Ralph Kimball).
  • ▶️ YouTube: Разбор Data Modeling (Star Schema, Snowflake, Data Vault 2.0).
