📅 День 1: Аналитические и оконные функции

Доступное время

4 часа
13:00 - 17:00 (Понедельник)

Теория "под капотом"

Физика оконных функций: Оконные функции (Window Functions) вычисляют агрегированные значения без схлопывания строк (как это делает `GROUP BY`). PostgreSQL разбивает данные на партиции (наборы строк) по `PARTITION BY`, сортирует внутри партиции по `ORDER BY`, буферирует их в `work_mem` (если памяти не хватает — сбрасывает на диск во временные файлы), а затем применяет функцию.

Frame Boundaries (ROWS vs RANGE): Окно (frame) определяет, какие строки попадают в вычисление для конкретной текущей строки. `ROWS` работает с физическими смещениями (например, ровно 3 строки назад), а `RANGE` — с логическими значениями (все строки с таким же значением сортировки). По умолчанию используется `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`, что может привести к O(n^2) при повторяющихся ключах сортировки и большом потреблении памяти.

RANK vs DENSE_RANK vs ROW_NUMBER: На уровне алгоритма: ROW_NUMBER() просто инкрементирует счетчик (O(n) после сортировки). RANK() увеличивает значение, но при совпадении ключа оставляет пропуски. DENSE_RANK() пропусков не делает.

Лайв-кодинг: Интерактивная практика

Задача 1: Топ-3 стрима по категориям
У нас есть таблица streams (stream_id, category, views). Напишите SQL-запрос, который найдет топ-3 самых популярных стримов в каждой категории с использованием оконной функции. Обратите внимание, что могут быть стримы с одинаковым количеством просмотров — используйте функцию, которая не пропускает ранги.
Подсказка 1: Поскольку требуется "не пропускать ранги" при совпадении количества просмотров, используйте DENSE_RANK(), сортируя по views DESC внутри каждой категории (PARTITION BY category).
Подсказка 2: Оконные функции нельзя использовать напрямую в секции WHERE. Оберните запрос с вычислением ранга в CTE (WITH) или подзапрос, а затем отфильтруйте во внешнем запросе WHERE rank <= 3.
WITH RankedStreams AS ( SELECT stream_id, category, views, DENSE_RANK() OVER (PARTITION BY category ORDER BY views DESC) as rank FROM streams ) SELECT stream_id, category, views FROM RankedStreams WHERE rank <= 3;

📅 День 2: Скользящие окна (Moving Average)

Доступное время

5 часов
09:00 - 14:00 (Вторник, нечетная неделя)

Теория "под капотом"

Определение границ фрейма: Скользящее среднее (Moving Average) критически зависит от правильного определения рамок кадра. Конструкция ROWS BETWEEN X PRECEDING AND Y FOLLOWING жестко фиксирует количество строк, участвующих в агрегации вокруг текущей строки.

Производительность агрегации: Вычисление AVG() или SUM() по скользящему окну требует, чтобы БД хранила в памяти все строки фрейма. Если фрейм очень большой, используется spilling на диск. Оптимизированные движки могут пересчитывать сумму, просто вычитая старый элемент, выпавший из фрейма, и прибавляя новый.

Лайв-кодинг: Интерактивная практика

Задача 2: 7-дневное скользящее среднее
Таблица daily_metrics (calc_date, daily_views). Рассчитайте 7-дневное скользящее среднее для просмотров (включая текущий день и 6 предыдущих). Данные отсортированы по дате.
Подсказка 1: Вам нужно использовать AVG(daily_views) OVER (...). Внутри OVER нужно задать порядок (ORDER BY calc_date) и ограничение строк: ROWS BETWEEN 6 PRECEDING AND CURRENT ROW.
SELECT calc_date, daily_views, AVG(daily_views) OVER ( ORDER BY calc_date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) as moving_avg_7d FROM daily_metrics ORDER BY calc_date;

📅 День 3: Рекурсивные CTE (Иерархии)

Доступное время

4.5 часа
08:30 - 11:00, 17:00 - 19:00 (Среда, нечетная неделя)

Теория "под капотом"

Механика RECURSIVE: Рекурсивные запросы (WITH RECURSIVE) не используют "рекурсию" в классическом понимании вызова функции самой себя. Это итеративный процесс под капотом.

Алгоритм: 1. Выполняется нерекурсивная часть (базовый якорь). 2. Результат помещается в рабочую таблицу (working table). 3. Рекурсивная часть выполняется на основе рабочей таблицы. 4. Новые сгенерированные строки помещаются в промежуточную таблицу (intermediate table). 5. Рабочая таблица очищается, данные из промежуточной становятся новой рабочей таблицей. 6. Цикл повторяется, пока промежуточная таблица не станет пустой.

Опасность циклов: Если граф имеет циклы (сотрудник A начальник B, а B начальник A), рекурсия зациклится. В современных версиях SQL используют CYCLE клаузу для предотвращения.

Лайв-кодинг: Интерактивная практика

Задача 3: Иерархия сотрудников
Таблица employees (emp_id, name, manager_id). Топ-менеджер имеет manager_id IS NULL. Напишите рекурсивный CTE, который выведет всех сотрудников и их уровень (depth) в организации. Топ-менеджер имеет уровень 0.
Подсказка 1: Внутри WITH RECURSIVE первая SELECT-часть должна выбирать корневого узла (где manager_id IS NULL) и жестко задавать колонку 0 as depth.
Подсказка 2: После UNION ALL присоедините таблицу employees e к вашей CTE через INNER JOIN org_chart oc ON e.manager_id = oc.emp_id, и увеличивайте глубину (oc.depth + 1).
WITH RECURSIVE org_chart AS ( -- Базовая (нерекурсивная) часть SELECT emp_id, name, manager_id, 0 as depth FROM employees WHERE manager_id IS NULL UNION ALL -- Рекурсивная часть SELECT e.emp_id, e.name, e.manager_id, oc.depth + 1 FROM employees e INNER JOIN org_chart oc ON e.manager_id = oc.emp_id ) SELECT * FROM org_chart ORDER BY depth, emp_id;

📅 День 4: EXPLAIN и стоимость запросов

Доступное время

4 часа
13:00 - 17:00 (Четверг, нечетная неделя)

Теория "под капотом"

Планировщик (Query Planner): В PostgreSQL используется стоимостной (cost-based) оптимизатор. Он строит множество возможных планов выполнения (деревья операций) и оценивает их "стоимость" (cost) в абстрактных единицах (обычно привязанных к 1 последовательному чтению страницы = 1.0).

Как читать EXPLAIN: Узел cost=startup_cost..total_cost rows=... width=.... Startup cost — время до выдачи первой строки. Total cost — время до выдачи всех строк. Сортировка всегда имеет высокий startup cost, так как БД обязана прочитать все данные, прежде чем отсортировать и выдать первую строку.

Nested Loop vs Hash Join vs Merge Join: Nested Loop хорош для малых выборок. Hash Join строит хеш-таблицу меньшей таблицы в памяти (быстро, но требует памяти). Merge Join требует отсортированных данных на входе.

Лайв-кодинг: Интерактивная практика

Задача 4: Оптимизация запроса с помощью индекса
Запрос SELECT * FROM users WHERE status = 'active' AND last_login > '2023-01-01' делает медленный Seq Scan. Предложите SQL-команду для создания комбинированного B-tree индекса (назовем его idx_users_status_login), который позволит планировщику использовать Index Scan.
Подсказка: Порядок колонок в комбинированном индексе критичен. Сначала ставьте колонки с равенством (status = 'active'), а затем колонки с диапазонами (last_login > ...). Значит, порядок (status, last_login).
CREATE INDEX idx_users_status_login ON users (status, last_login);

📅 День 5: Индексы и B-tree

Доступное время

4 часа
15:00 - 19:00 (Пятница)

Теория "под капотом"

Структура B-Tree (Сбалансированное дерево): Страницы (pages) размером 8KB разбиты на корневые узлы, внутренние и листовые. Поиск значения — спуск от корня к листу. Это дает алгоритмическую сложность O(log N). Из-за высокого fanout (количества детей у узла), база в 1 млрд записей имеет дерево высотой всего 4-5 уровней.

Index-Only Scan: Если запрос запрашивает только те колонки, которые уже содержатся в самом индексе, база может не обращаться к основной таблице (heap). Для этого данные в heap должны быть видимы (проверяется через Visibility Map). Это один из самых мощных способов оптимизации.

Partial Indexes (Частичные): Индекс можно создать только для подмножества данных `WHERE deleted_at IS NULL`. Это экономит место на диске, повышает плотность в кэше RAM и ускоряет запись.

Лайв-кодинг: Интерактивная практика

Задача 5: Частичный покрывающий индекс
Мы часто выполняем: SELECT email FROM users WHERE is_active = true;. Напишите код создания индекса (назовите его idx_active_users_email), который будет частичным (только для active) и будет поддерживать Index-Only Scan (должен хранить email).
Подсказка 1: В PostgreSQL для "покрывающих" индексов (Covering index), если колонка не участвует в поиске/фильтрации, её лучше добавить через секцию INCLUDE (email), а затем поставить условие частичности WHERE (is_active = true). Либо просто сделать классический композитный и частичный ... ON users (email) WHERE is_active = true.
CREATE INDEX idx_active_users_email ON users (email) WHERE is_active = true; -- Альтернативно: ... ON users (is_active) INCLUDE (email) WHERE is_active = true;

📅 День 6 и 7: Транзакции, MVCC и UPSERT

Доступное время

11 часов
Сб: 09:00-11:00, 19:00-22:00 | Вс: 10:00-16:00

Теория "под капотом"

MVCC (Multi-Version Concurrency Control): В PostgreSQL при выполнении UPDATE старая строка не затирается. Вместо этого старая строка помечается как "удаленная" в этой транзакции (проставляется xmax), а новая вставляется как актуальная. Читающие процессы продолжают читать старую строку, пока транзакция не закоммитится. Очистку старых ("мертвых") строк проводит процесс VACUUM.

Уровни изоляции и аномалии: Read Committed защищает от "грязного чтения", но допускает Phantom Reads и Non-Repeatable Reads. Если вы делаете два SELECT в одной транзакции, в разных SELECT результат может меняться. Если нужна полная безопасность данных при каскадной записи (например банкомат) — используют Serializable или блокировки на уровне строк SELECT ... FOR UPDATE.

Лайв-кодинг: Интерактивная практика

Задача 6: Идемпотентный UPSERT (Вставка с обновлением)
Вам нужно вставить нового пользователя `INSERT INTO users (id, name, visits) VALUES (1, 'Alice', 1)`. Но если пользователь с таким id (первичный ключ) уже существует, нужно просто увеличить поле visits на 1. Реализуйте этот запрос для PostgreSQL.
Подсказка 1: Используйте синтаксис ON CONFLICT (id) DO UPDATE SET ..., а для обращения к старому значению используйте префикс таблицы users.visits + 1.
INSERT INTO users (id, name, visits) VALUES (1, 'Alice', 1) ON CONFLICT (id) DO UPDATE SET visits = users.visits + 1;
