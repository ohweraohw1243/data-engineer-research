🐘 Этап 1: Перевод сырых логов в витрины (DWH)

📂 GitHub Repo: e-commerce-dwh-modeling

Бизнес-задача: У вас есть выгрузки "грязных" логов покупок интернет-магазина (в CSV) и справочники клиентов. Нужно построить хранилище в PostgreSQL по методологии Кимбалла (Star Schema) и написать SQL-скрипты миграции (ETL).

План действий:

  • Сгенерировать фейковые данные (мокап) с помощью Python (Faker) или скачать датасет с Kaggle. Сохранить как raw_events.csv.
  • Поднять локально Docker-контейнер с PostgreSQL.
  • Написать DDL-скрипты для слоя STG (Stage), где данные лежат "как есть" (типы text/varchar).
  • Спроектировать DDS слой (Dimensional Data Store): таблицу фактов fact_purchases и измерения dim_users, dim_products. Реализовать SCD Type 2 (сохранение историчности изменений у клиентов).
  • Написать SQL(PL/pgSQL) скрипты/функции или просто `.sql` файлы, которые делают `INSERT INTO ... SELECT` для переливки данных STG -> DDS.
🌟 Бонус (Performance)

Сгенерировать 10 млн+ строк. Сделать запросы к таблице фактов медленными. Проанализировать EXPLAIN ANALYZE. Создать правильные B-Tree / BRIN индексы, применить секционирование (Partitioning) таблицы фактов по дате покупки и доказать, что запрос ускорился.

🐍 Этап 2: Custom Python ETL Framework & S3 Upload

📂 GitHub Repo: external-api-to-datalake-etl

Бизнес-задача: Данные лежат в стороннем REST API (например, API погоды, криптобиржи Binance или Github API). Задача — надежно их выкачивать каждый день, парсить JSON, валидировать и складывать в Data Lake (S3).

План действий:

  • Поднять локально MinIO (аналог Amazon S3) в Docker.
  • Написать ООП-класс-коннектор на Python (с использованием requests, aiohttp), который забирает данные по API с учетом пагинации (pagination) и Rate Limits (использовать backoff/retry декораторы).
  • Использовать библиотеку Pydantic или Pandera для строгой валидации JSON (своя Schema-on-Write проверка).
  • Преобразовывать (flatten) сложный JSON в плоскую структуру с помощью Python (pandas/boto3/pyarrow) и сохранять локально в формат Parquet.
  • Написать логику загрузки Parquet-файла в MinIO локально в бакет s3://raw-zone/binance/year=2026/month=04/ (партиционирование по датам в стиле Hive).
🌟 Бонус (Асинхронность и ООП)

Реализовать скачивание данных асинхронно через asyncio и aiohttp. Завернуть логику выгрузки в базовый абстрактный класс `BaseExtractor`, от которого будут наследоваться экстракторы для разных API (соблюсти SOLID).

⚡ Этап 3: PySpark Big Data Processing

📂 GitHub Repo: spark-datalake-pipeline

Бизнес-задача: Нужно обработать "огромный" объем сырых данных (логи посещений веб-сайта, кликстрим), очистить их, присоединить к справочникам и собрать аналитическую витрину активности пользователей (сессии). Объем такой, что Pandas уже падает из-за нехватки RAM.

План действий:

  • Написать Python-скрипт для генерации "тяжелого" датасета (10+ ГБ).
  • Поднять локальный Spark (Pyspark через Jupyter/Zeppelin или просто python-скрипты, запускаемые через `spark-submit`).
  • В PySpark прочитать данные (CSV/JSON), выполнить очистку (удаление дублей, fillna, приведение типов, парсинг User-Agent или URL).
  • Выполнить Broadcast JOIN с маленькой таблицей справочника (например, `dim_devices`). Сделать агрегаты (Group By) с использованием Window Functions (найти первую и последнюю страницу пользователя в сессии).
  • Сохранить результат (Data Mart) как Parquet с секционированием. Показать разницу в объеме (до и после сжатия Snappy в Parquet).
🌟 Бонус (Оптимизация и Внутренности Spark)

Специально спровоцировать проблему Data Skew (перекоса данных) (например, 90% данных относятся к одному device_id=NULL). Показать через Spark UI, что одна таска висит бесконечно долго. Решить проблему через Salting (добавление соли к ключам JOIN) или фильтрацию до JOIN, и задокументировать это скриншотами.

🌊 Этап 4: Real-time Streaming (Kafka + ClickHouse)

📂 GitHub Repo: realtime-fraud-detection-streaming

Бизнес-задача: Магазин хочет бороться с мошенниками в реальном времени. Если с одного аккаунта за 1 минуту поступает более 5 транзакций из разных городов — транзакцию нужно заблокировать (оповестить).

План действий:

  • Поднять мощный `docker-compose.yml`, в котором работают: Zookeeper, Kafka, ClickHouse.
  • Написать Producer на Python, который бесконечно генерирует фейковые события транзакций (JSON с `user_id`, `amount`, `city`, `timestamp`) и шлет их в топик Kafka transactions_raw.
  • В ClickHouse настроить движок таблиц Kafka Engine, который будет читать данные напрямую из топика и перекладывать через Materialized View в таблицу MergeTree (или ReplacingMergeTree).
  • Продемонстрировать способность ClickHouse молниеносно агрегировать данные и рисовать график (например, подключить туда Metabase или Superset "на коленке", или просто SQL-запросами показать fraud-операции).
🌟 Бонус (Spark Structured Streaming)

Вместо `ClickHouse Kafka Engine` написать пайплайн на Spark Structured Streaming. Он читает топик `transactions_raw`, выполняет расчет агрегата с окном (Tumbling Window в 1 минуту), записывает suspicious аккаунты в другой (output) топик fraud_alerts.

🏗️ Этап 5: Оркестрация с Airflow & Финальная сборка

📂 GitHub Repo: end-to-end-airflow-datalake

Бизнес-задача: У вас есть весь написанный ранее код (забор по API, SQL миграции в PostgreSQL, Spark джобы), но чтобы оно работало автоматически, его нужно "оркестрировать" — засунуть в планировщик, поставить зависимости и настроить алерты об ошибках.

План действий:

  • Развернуть официальный Apache Airflow через docker-compose (с CeleryExecutor и Redis).
  • Написать 1 комплексный DAG, который объединяет все шаги пайплайна:
    1. Sensor - ждет появления файла/события.
    2. PythonOperator - забирает данные по API и кладет в MinIO (Этап 2).
    3. SparkSubmitOperator или DockerOperator - запускает PySpark job для очистки и сохранения витрины (Этап 3).
    4. PostgresOperator - обновляет агрегаты или переливает очищенные витрины в базу (Этап 1).
  • Настроить обработку ошибок (on_failure_callback) с отправкой уведомления в Telegram-бота.
  • Использовать XCom или передачу параметров через S3 (не гонять тяжелые данные через XCom!).
  • Подготовить классный README.md в репозитории с архитектурной схемой пайплайна (нарисованной в draw.io) и прикрепить его в резюме!
