Чем Hash Join отличается от Merge Join? ▼
Ответ:

Hash Join O(n+m): Строит хеш-таблицу по образцу (наименьший набор), затем сканирует дорожный набор. Требует O(n) памяти, но очень быстро.

Merge Join O(n log n + m log m): Требует отсортированные наборы. Работает как zipper — идет по обоим потокам одновременно.

Выбор: Hash для маленьких inner tables, Merge для sorted input, Nested Loop для маленьких наборов с индексом.
Как работает MVCC в PostgreSQL? xmin/xmax? ▼
Ответ:

MVCC: Каждая строка имеет версии. Читатели берут snapshot транзакций на момент начала.

xmin/xmax:
• xmin — ID транзакции, создавшей версию
• xmax — ID транзакции, удалившей версию

SELECT проверяет видимость: если xmin видна и xmax не видна, строка живая. VACUUM очищает версии когда все xmax < active xids.
B-tree vs Hash индексы ▼
Ответ:

B-tree: Диапазонные запросы, сортировка, height = log_fanout(N).

Hash: Только точное совпадение O(1), не поддерживает диапазоны и NULL.

Практика: В 99% B-tree лучше. Hash редко используется.
Isolation levels и trade-offs ▼
Ответ:

Read Committed (default): Видит committed данные. Быстро, но некоторые anomalies.

Repeatable Read: Snapshot на BEGIN. Защищает от lost updates.

Serializable: Полная изоляция. PostgreSQL использует SSI. Может быть slow и требует retries.
multiprocessing vs threading в Python ▼
Ответ:

Threading: Один процесс, несколько потоков. GIL — только один bytecode выполняется. I/O-bound OK, CPU-bound slow.

Multiprocessing: Отдельные процессы, истинный параллелизм. Overhead: pickle, memory duplication. CPU-bound OK.

Правило: I/O → threading, CPU → multiprocessing, I/O+compute → asyncio.
Как работает GIL и почему он существует? ▼
Ответ:

GIL: Мьютекс, защищающий Python объекты. Один потокзащитает bytecode одновременно.

Зачем: Reference counting (rc) рискует race conditions. Вместо fine-grained locking всех объектов используется один глобальный lock.

Последствия: CPU-bound threading не ускоряет. I/O-bound работает (GIL отпускается при I/O).
async/await на низком уровне ▼
Ответ:

async def: Возвращает coroutine object. Не выполняется сразу.

await: Выдает управление event loop. Если operation не готова, loop переходит к другой coroutine. Когда готова — resumes.

Event loop: Кооперативная многозадачность. Много coroutines могут быть pending I/O одновременно. CPU-bound блокирует весь loop.
Backpressure и как с ним справляться ▼
Ответ:

Producer быстрее consumer → queue переполняется, memory растет.

Решения:
1. Buffer с лимитом: producer блокируется когда полнка
2. Semaphore: ограничивает одновременные задачи
3. Rate limiting: замедляет producer
4. Sampling: берет только часть элементов
SCD Type 1 vs Type 2 ▼
Ответ:

Type 1 — Overwrite: Просто обновить старую запись. Быстро, теряется история.

Type 2 — History: Новая строка с is_current флагом и valid_from/valid_to. История сохраняется. Медленнее и сложнее.
Exactly-once семантика в ETL ▼
Ответ:

Семантики:
• At-most-once: потеря возможна
• At-least-once: дубли возможны
• Exactly-once: каждое событие обрабатывается один раз

Техники:
1. Idempotency keys: UUID для каждого события
2. Deduplication: кешировать processed_keys
3. Idempotent ops: SET vs UPDATE +=
4. exactly-once source + idempotent sink + duplicate detection
